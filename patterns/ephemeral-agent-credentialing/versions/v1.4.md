## 1. Executive Summary

An AI agent is not a service that persists. Every user action, every workflow trigger, every inbound event spawns a new agent instance that asks an LLM what to do, does one task, and ends. The same workflow triggered twice is two different agents — the LLM re-plans the path on every run, touching different systems in different orders depending on what the model decides in that moment. That runtime nondeterminism is the feature that makes agents useful. It is also the reason identity and access management models built for humans and services do not fit them.

Most enterprise IAM was designed for humans who log in once a day and for services whose behavior is defined before they ship. Credentials are granted broadly, rotated slowly, and scoped to what a workload *might* need rather than what it actually does on any given run. For traditional services that model works. For agents it breaks at the moment of first invocation, because there is no single "agent" to credential — there is a continuous stream of short-lived instances, each one born, credentialed, and retired.

The result in practice is an operational pattern that looks something like this: a single workflow fires ten thousand times a day. Every invocation is a new agent with a new runtime-decided path. All ten thousand authenticate with the same long-lived credential. The agent ends when its task ends — seconds to minutes. The credential does not. It lives on, valid, unused, and waiting.

That gap — between when the agent ends and when the credential ends — is where the exposure lives. A credential stolen mid-task is still valid minutes after that task is done, on a different path, for a different agent that has not yet been born. Attribution is impossible in this model: every action in the logs traces back to the shared credential, not to the specific agent instance that performed it.

This framework proposes a different posture for agent credentials specifically: issue a fresh, cryptographic identity to *every* agent instance at spawn time, scope its access to the task in front of it, validate every request, expire credentials when the task ends, and record each action to a tamper-evident log that attributes the action to that specific instance.

This is not an argument to rebuild the identity stack. Existing IAM remains the right home for directories, policy engines, gateways, and federation. The argument is narrower: the individual agent instance needs a credential model that matches how it actually operates — ephemeral, LLM-planned, one-task-one-lifetime.

**What adoption provides.**
A sharply reduced credential exposure window. Per-agent attribution for every action. Direct, structural alignment with GDPR data minimization, HIPAA unique-identifier requirements, PCI-DSS 4.0 non-human identity controls, and SOC 2 evidence expectations. Faster incident containment. A credible answer for auditors asking how autonomous systems are governed.

**What the framework is not.**
It is not a replacement for existing identity providers, policy engines, or governance programs. It is not a promise of zero risk. For organizations standing up agents now, adoption is a design choice made at first deployment. For organizations already running agents at scale, adoption is a workflow-level migration: each workflow, one at a time, moved to per-instance identity and task-scoped credentials. The object of governance is the workflow, not the agent instance — the agent does not exist outside a workflow.

## 2. The Business Problem in Plain Terms

### 2.1 Every user action spawns a new agent. The LLM decides the path at runtime.

The most useful reframe for this framework is the definition of the word "agent." Most executives hear "AI agent" and picture something like a service — one running thing that handles requests. That picture is wrong, and the wrongness is where the exposure hides.

Here is what is actually happening. An orchestrator receives a trigger — a user clicks a button, an event fires, a scheduled job starts. The orchestrator spawns an agent. That agent asks an LLM what to do. The LLM reads the task, decides which tool to call first, which data to read, which step to take next. The agent executes that step, asks the LLM again, executes the next step, and so on until the task is done. Then the agent ends. Its process exits. Its memory is gone. Its identity — if it had one at all — is retired.

The next trigger spawns a new agent. The LLM is called again, from scratch. It will not necessarily make the same decisions. Given near-identical inputs, the same workflow can query the CRM first on Monday and the wiki first on Tuesday. It can touch three systems on one run and seven on the next. That unpredictability is not a bug. It is the whole reason the organization chose an agent over a script.

**An analogy that tends to land in executive rooms.** Think of a 500-room hotel. "Housekeeping" is one service on paper — one team, one workflow, one master key. In reality, two hundred housekeepers walked through your hotel today. Different people, different rooms, different decisions about which rooms to enter and in what order. They each used the same master key. If one of them lost that key mid-shift, it is not one room that is exposed. It is every room, every shift, every floor, for as long as that master key has existed. The agent workflow is "housekeeping." Each invocation is a different housekeeper. The credential is the master key. Today you had ten thousand housekeepers. All on the same key.

That is the structural break under every static credential model. Credentials are issued at deploy time, to a workload. But there is no workload — there is a stream of short-lived instances, each one deciding its own path through the environment. The organization is left with two uncomfortable choices:

- Grant broad credentials to cover every possible path the LLM might take.
- Grant narrower credentials and track what accumulated later.

The first is risk acceptance dressed as convenience. The second is a cleanup operation dressed as strategy.

### 2.2 The exposure math is not subtle.

Consider a conservative production profile. One hundred concurrent agent instances running at any moment. Each task takes two minutes. Each instance authenticates with a standard fifteen-minute OAuth token. The organization runs one thousand task cycles a day — one thousand births, one thousand deaths, one thousand LLM-planned paths through the environment.

In that system, on any given day, roughly twenty-one thousand agent-hours of credential lifetime exist outside the window of actual work. The agents are long gone. The credentials are still valid. A credential stolen at minute fourteen of the Monday-morning agent is still useful at minute twelve of a completely different agent running a completely different task that afternoon.

Shortening credential lifetime to the task itself collapses that exposure window to near zero. The credential dies when the agent dies. The math is not an argument by itself. It is the shape of the opportunity.

### 2.3 Regulators are already asking the right questions.

Compliance frameworks written for human and service identities are being applied to agent workloads in 2026. GDPR Article 5 requires data access to be limited to what is necessary for a specific purpose. HIPAA §164.312 requires unique identifiers for anyone accessing protected health information. PCI-DSS 4.0 Requirement 8.6 adds explicit controls for non-human identities. SOC 2 expects an evidence trail linking actions to actors.

Each of those controls lands awkwardly on a shared service account or a long-lived bearer token. Each of them lands cleanly on a model where every agent instance has its own identity and every credential has a lifetime matched to the task.

The question is no longer whether auditors will notice the gap. The question is how defensible the answer will be when they do.

## 3. Why Extending Existing IAM Is Not Enough

A number of well-respected frameworks are now emerging to address agent identity. Several of them take the thoughtful position that agents should be brought into existing IAM rather than managed outside it. They add claims for non-human principals, extend token exchange patterns for on-behalf-of delegation, define capability tiers, and describe gateway enforcement models.

That work is real and it is valuable. Enterprises will extend their existing IAM stacks rather than replace them. Governance, federation, and policy belong in the control plane that is already there.

The concern with this framing — not the work itself, but the framing — is that it can be read to suggest that agent identity is primarily an extension problem. That if enough claims are added to existing tokens, enough gateway policies are written, and enough tiers are defined, the problem is solved.

The fifty-thousand-foot view is that agents are not close enough to traditional services for extension alone to carry the weight. The break is not the vocabulary. It is the assumption that access can be decided in advance for a workload whose path is decided at runtime.

A more honest framing separates two concerns:

- **Governance belongs at the operator and orchestrator boundary.** Who is allowed to launch agents? What ceilings apply to each application? Which models and policies are approved? These are questions existing IAM is well-suited to answer.
- **Credentialing belongs at the agent instance boundary.** What identity does this specific agent carry on this run? What scope does it hold for this task? When does its credential expire? These are questions that require a different posture than static-assignment IAM was built for.

This framework is about the second boundary. It does not dismiss the first. It is, deliberately, a narrow claim: the individual agent instance needs a credential model that reflects runtime, task-scoped, non-deterministic behavior.

## 4. The Framework: Five Principles Leadership Should Recognize

The full pattern defines eight technical components. For an executive audience, the posture resolves into five plainer principles.

### 4.1 Every agent instance gets its own identity, once

When an orchestrator spawns an agent for a task, it receives a unique cryptographic identity that encodes which orchestrator launched it, which task it is serving, and which specific instance it is. When the task completes, that identity is retired. Compromise of one agent cannot grant access to another because no two agents share an identifier.

### 4.2 Credentials are scoped to the task, not the session

Tokens are issued for the specific resources and actions the task requires, and for a lifetime that matches how long that work should take. Long-running tasks renew their tokens in place, but renewal cannot expand scope. The credential ends when the work ends.

### 4.3 Every request is re-validated

No request is trusted because a prior request was trusted. Every call re-establishes identity, scope, expiration, revocation status, and, where applicable, delegation lineage. This is the zero-trust posture applied to agent traffic specifically.

### 4.4 Every action is attributable, forever

Every credential operation, every tool call, every delegation, every revocation is written to a tamper-evident, append-only log. The log is designed so that a forensic question — which agent accessed which record on whose authority between these two timestamps — becomes a query, not an investigation.

### 4.5 Governance at the boundary, credentialing at the instance

Applications, orchestrators, and users are governed at the control plane. They have registered identities, approved policies, ceilings on what they may grant, and regulated paths into the agent runtime. The individual agent instance inherits none of that standing authority. It is credentialed at runtime, for this task, under those ceilings, and nothing more.

These five principles are what a board member should be able to recognize. The technical implementation is worked out in the companion v1.4 (Technical Edition) document through eight detailed components. The executive posture above is what the implementation is trying to produce.

## 5. The Five Risks That Matter Most Right Now

Seventeen risks in the full threat model are scored critical. The five below are the ones with the most direct executive consequence in 2026 — the ones most likely to become a breach disclosure, an audit finding, or a board conversation.

### 5.1 Prompt injection turning into credential theft

An attacker does not have to break the infrastructure. They submit a support ticket, a document, or a web page containing crafted text. The agent processes it, and if it is holding static credentials in memory or environment variables, a well-formed injection can cause those credentials to be exposed in the agent's response or logs. The attacker now holds real credentials.

Under the ephemeral model, the same injection returns only a short-lived token scoped to the current task. The attack compresses from "persistent credential theft" to "five-minute single-task scope." The attacker keeps a useless artifact.

### 5.2 One credential behind thousands of invocations

The common pattern is a single service account deployed with an agent workflow, authenticating every instance the workflow ever spawns. In a typical production environment that is not tens of agents sharing one credential — it is tens of thousands of invocations per day, every one of them a fresh agent making its own runtime-decided path through the environment, all reporting as the same identity to every downstream system.

**The contractor-badge analogy.** Imagine every contract worker who walks into your building today swipes the same shared "CONTRACTOR" badge. There were four hundred of them — different people, different floors, different reasons for being there. When something goes wrong — a missing laptop, an unauthorized access to a sensitive area — the badge log says "CONTRACTOR" four hundred times. You cannot tell which contractor did what. And because the badge is shared, you cannot revoke it without locking out the other three hundred and ninety-nine.

That is the real-world shape of the risk. A compromise at any one invocation leaks a credential that every subsequent invocation will also use. Attribution is structurally impossible: every log line shows the shared identity, not the specific agent instance that performed the action. Revocation breaks the entire workflow, not just the compromised run.

Per-instance identity closes both failures at once. Compromise is bounded to the scope of a single task. Every action is attributable to a specific agent instance. Revoking a compromised instance has no impact on any other.

### 5.3 Long-lived credentials and GDPR data minimization

GDPR Article 5(1)(c) requires that data access be limited to what is necessary for a specific purpose. A fifteen-minute token attached to a two-minute task is not that. In an audit, long-lived credentials granted to agents that process personal data are increasingly read as a direct compliance finding, not a grey area.

Task-scoped credentials provide direct, auditable evidence that access was bounded to purpose. The pattern produces a compliance artifact by default.

### 5.4 Non-attribution during regulatory reporting

Most breach notification frameworks require the organization to identify which records were accessed, by whom, and during what window. When agents operate under shared accounts or broadly-scoped credentials, that information does not exist in a defensible form. The organization then faces a choice between over-disclosure and regulatory exposure.

Per-agent audit logs with hash-chained integrity make that conversation a log query. Forensic reconstruction stops being guesswork.

### 5.5 Supply chain credential exposure

An agent loads a dependency, calls a third-party tool, or integrates an MCP server. That dependency runs in the agent's context. If the agent carries static credentials, those credentials are now available to anything the dependency can reach, including anything compromised upstream in the supply chain.

The ephemeral model makes the blast radius of a supply chain compromise proportional to a single task, not a whole environment. The dependency gets a minute-scale token, not a long-lived key.

These five are not the entire risk picture. They are the five a risk committee should be able to recognize in plain language. Appendix A summarizes the full set of seventeen critical risks identified in the threat model.

## 7. The Window Is Now

Agent identity has been a recognized problem for less than 18 months. Most organizations are in their first year of serious agentic deployment. The technical debt in AI agent credentials has not been fully written yet. That is the window.

Organizations standing up production agents today can design the credential model to match how agents actually work — non-deterministic, ephemeral, runtime-scoped — rather than forcing agents into an identity model built for a fundamentally different kind of workload. Build this correctly the first time, and the phased remediation roadmaps other frameworks offer never become necessary.

The wrong paths are already visible. Broad credentials that cover every possible path an agent might take are risk acceptance dressed as convenience. Systems that track and govern the entitlements agents accumulate over time are cleanup operations dressed as strategy. Both concede that the credential shape is wrong, then invest in making the wrong shape auditable.

The five principles in this framework — per-instance identity, task-scoped credentials, every request re-validated, every action attributable, governance at the boundary — are design choices made at the point of first agent deployment. They are not milestones on a remediation roadmap, and they do not require a visibility-and-assessment phase to precede them. The thing that persists under this framework is the orchestrator and the credential issuer. Those are in scope for governance the day they are deployed. The agents themselves never persist long enough to inventory.

For organizations already running production agents on shared credentials, the move is not to inventory agents. Agents do not exist outside the workflow that invokes them; every instance ends when its task ends. What persists — and what should be inventoried — is the workflow. The trigger, the orchestrator, the shared credential, the downstream integrations. Each of those is a discrete, governable object. The work is a workflow-level migration: take each workflow one at a time, rebind it to per-instance identity and task-scoped credentials, and cut over the shared credential that every future invocation would have ridden. Migration, not remediation — because the thing being migrated is the workflow, not an accumulated pool of agents that no longer exist. The right time to get this correct is before the mess has been made; for the workflows already at scale, the honest move is to migrate the workflow itself, not to track what accumulated under the wrong shape.

## 8. Governance and the Existing IAM Investment

This framework is not a parallel IAM stack. It is deliberately designed to sit on the rails already in place.

**What remains in the existing control plane.**
Directories, identity providers, OAuth and OIDC servers, policy engines, PKI, secrets management, and SIEM tooling all continue to operate at the governance layer. Who is allowed to operate an orchestrator, which applications are approved, what policies apply at what ceiling, how cross-domain trust is expressed, where audit data is aggregated — these questions remain where they belong.

**What is added at the instance layer.**
A credential issuance service that produces per-instance, task-scoped credentials on demand and at runtime. A validation path that checks identity, scope, expiration, and revocation on every request. A tamper-evident audit log that attributes every action to the specific agent instance that performed it.

**What this separation preserves.**
The enterprise does not rebuild its identity stack to adopt agents. It extends the governance layer it already invested in, and adds a new, narrowly-scoped component at the place where existing IAM was never designed to reach.

This is the layered model. Governance above the agent. Credentialing at the agent.

## 9. What Executives Should Ask Their Teams

The right questions clarify posture faster than any framework diagram.

- **How many distinct identities do our agent workflows use, and how many invocations per day run under each one?** If any identity is handling more than one invocation at a time, the organization is operating on shared-credential assumptions. The right answer is one identity per instance, retired when the instance ends.
- **What is the median credential lifetime held by an agent instance, and what is the median task duration?** The ratio between those two numbers is the exposure window. A fifteen-minute token on a two-minute task is a seven-times-multiplier of credential life over useful life.
- **If one agent instance is compromised right now, what is the blast radius?** If the answer extends beyond that single instance's task, the credential model is the reason — the compromise leaked a credential every other instance also uses.
- **For any action taken by an agent in the last thirty days, can we produce the agent instance, task, authorizing principal, and delegation chain within fifteen minutes?** If not, regulatory reporting is at risk.
- **Are any agents accessing regulated data — personal data, protected health information, payment data — under shared accounts or long-lived credentials?** If so, that is the first population to move to the ephemeral profile.
- **Of the agents we plan to ship in the next two quarters, how many will launch under per-instance identity and task-scoped credentials from day one?** If the answer is anything less than all of them, we are writing technical debt into a credential layer that is still new enough to get right the first time.
